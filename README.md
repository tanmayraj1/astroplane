# Astroplane

**The daily planner that knows your sky.** Wake windows, power hours, tarot and
gentle nudges — cast fresh from your real birth chart every morning.

Built with Next.js 16 (App Router), Supabase, OpenAI, Stripe + Razorpay, and a
pure-TypeScript astrology engine on `astronomy-engine` (real ephemeris math —
planets, whole-sign houses, transits, void-of-course Moon, planetary hours).

---

## What's inside

| Layer | What it does |
|---|---|
| `src/lib/astro` | Natal charts (whole-sign, solar fallback), aspects, daily transits, moon phase + void-of-course, planetary hours → power/friction windows, color of the day. Pure TS, portable to React Native. |
| `src/lib/ai` | OpenAI daily-card generator (engine data passes through — AI only phrases), tarot interpreter, crisis-language safety gate. Falls back to deterministic copy with no API key. |
| `src/lib/billing` | Dual provider: Stripe (international, USD) + Razorpay (India, INR). Subscriptions with 7-day trial, wallet top-ups, idempotent per-minute chat billing via SECURITY DEFINER RPCs. |
| `supabase/` | 5 migrations (full schema + RLS on every table) + seed (6 AI guide personas). |
| `src/app` | Landing, pricing, auth (email OTP + Google), onboarding ×4 with Big-Three reveal, Today, Planner (week grid + timeline), Chart (natal wheel SVG), Tarot (3D flip + journal), Guides + live per-minute chat, Patterns, Profile, Paywall, Billing, Wallet. Fully responsive: desktop sidebar shell + mobile floating glass tab bar. Installable PWA. |

## Setup (once)

### 1. Supabase
1. Create a project at [supabase.com](https://supabase.com).
2. In the SQL editor, run each file in `supabase/migrations/` **in order** (0001 → 0005), then `supabase/seed.sql`.
3. Auth → Providers: enable **Email** (OTP) and optionally **Google** (add OAuth credentials; set callback to `https://<project>.supabase.co/auth/v1/callback`).
4. Auth → URL configuration: add your site URL + `http://localhost:3000`.
5. Copy Project URL, anon key, service-role key into `.env.local`.

### 2. OpenAI
Create an API key at platform.openai.com → `OPENAI_API_KEY`. (Without it the app still runs — daily cards and tarot fall back to the deterministic rules engine, and guides reply with a "not configured" notice.)

### 3. Stripe (test mode first)
1. Create two recurring Prices: Plus $7/mo, Pro $16/mo → put their `price_...` ids in env.
2. Add a webhook endpoint `https://<your-domain>/api/webhooks/stripe` with events: `checkout.session.completed`, `customer.subscription.*`, `invoice.payment_failed` → copy the signing secret.

### 4. Razorpay (test mode; live needs Indian KYC)
1. Create two Plans: Plus ₹299/mo, Pro ₹899/mo → `plan_...` ids in env.
2. Add webhook `https://<your-domain>/api/webhooks/razorpay` with events `subscription.*`, `payment.captured`, `payment_link.paid` → set the webhook secret.
3. Keep `RAZORPAY_LIVE=false` until KYC clears — Indian users then fall back to Stripe.

### 5. Run
```bash
cp .env.example .env.local   # fill it in
npm install
npm run dev
```

### 6. Deploy (Vercel)
- Import the repo, add all env vars, set `NEXT_PUBLIC_APP_URL` to the production URL.
- `vercel.json` already schedules the two crons (daily cards hourly, session sweep every 5 min). Set `CRON_SECRET` in env — Vercel sends it automatically as the Authorization bearer.
- Point Stripe/Razorpay webhooks at the production domain.

## Architecture notes

- **The AI never invents astronomy.** All times, windows, colors and transits come from the deterministic engine; OpenAI only phrases them. Every AI surface carries the disclosure "Generated by AI Planner · trained by our astrologers".
- **Billing integrity:** money tables are service-role-only; wallet writes go through atomic `wallet_credit` / `wallet_debit_tick` RPCs with unique-constraint idempotency (`provider_payment_id`, `(session_id, minute_index)`). Duplicate webhooks and double heartbeats are structurally harmless. A cron sweep ends sessions whose heartbeat goes stale, so a closed tab can't keep billing.
- **Charts:** computed once at onboarding (recomputed if birth data changes), cached in `natal_charts`. The location-shared daily sky is cached per `(date, tz, 1° lat/lng bucket)` and shared across users.
- **Safety:** crisis language is pattern-detected *before* any model call and answered with region-appropriate helplines (AASRA/iCall/Tele-MANAS for India, 988/Samaritans elsewhere), never a card. Journals are never sent to AI without an explicit user action.
- **Timezones:** all "today" logic is per-user IANA timezone (luxon); planetary hours are true sunrise/sunset unequal hours.
